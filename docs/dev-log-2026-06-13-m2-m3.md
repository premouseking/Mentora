## 做了什么

### M2 — 数据契约

设计并实现了解析产物和证据单元的数据契约，作为 LH（资料与检索）在 Stage 1 的核心交付。

- **BoundingBox**：PDF 标准坐标系（左下角原点，pt 单位），带 x1>=x0 校验
- **ParsedElement**：6 种元素类型（paragraph/heading/table/formula/image/list_item），含文本、坐标、层级、置信度
- **Page**：页码（物理序 + 原文件标注）+ 元素列表（阅读顺序）
- **ParsedBundle**：一次完整解析的顶层容器，含 content_hash、parser 版本、质量评估、artifact_ref
- **EvidenceUnit**：可引用证据，通过 element_indices 引用 ParsedElement（不复制坐标数据）
- **EvidenceSnapshot**：模型调用时的不可变证据集合

所有模型为 Pydantic v2，支持 JSON Schema 导出供跨语言消费。坐标系、页码规则和幂等键策略写入 ADR-0006。

### M3 — PyMuPDF 解析 Worker

实现第一个解析器适配器，从 PDF 产出符合 M2 契约的 ParsedBundle。

- **PyMuPDFAdapter**：使用 `get_text("dict")` 提取结构化文本块，根据字体大小推断 heading/paragraph
- **evidence.py**：heading 与后续 paragraph 合并为一个 EvidenceUnit，跳过图片和空文本
- **tasks.py**：Celery 任务骨架，content_hash + parser_name + version 组成幂等键，3 次重试
- **异常分类**：EncryptedPDFError、CorruptedPDFError、ImageOnlyPDFError、UnsupportedFormatError
- **适配器注册表**：按文件扩展名路由，新增解析器只注册不修改调用方

### 测试

- 程序化生成 3 个 PDF Fixture（正常文本、含标题、模拟多栏）
- 6 项测试通过：解析基本功能、标题分类、JSON 往返、证据拆分、多页标注、图片跳过

## M5-B 追加：jieba + pg_trgm 检索方案（2026-06-13）

在 M5 Evidence ORM 持久化之前，先完成中文分词方案选型和检索原型。

### 方案选型结论

选择 **jieba + pg_trgm 双层策略**（同时排除 zhparser）。

| 方案 | 优势 | 劣势 | 采用 |
|---|---|---|---|
| pg_trgm | PG 内置、零安装、模糊容错 | 不识别词边界，术语噪声大 | 模糊层兜底 |
| jieba | CS 术语词典精准控制、Python 原生 | 需应用层分词后拼 tsquery | 精确层主力 |
| zhparser | PG 原生 FTS + 向量混合查询 | C 扩展安装维护成本高，活跃度低 | 不采用 |

### jieba 自定义词典

55 条 CS 术语，覆盖三门课程：
- 计算机组成原理：直接映射、组相联、Cache 行、TLB、指令流水线、中断向量表...
- 数据结构：B+ 树、AVL 树、红黑树、哈希表、二分查找...
- 机器学习：梯度下降、反向传播、卷积神经网络、过拟合、正则化...

### 核心模块

| 文件 | 说明 |
|---|---|
| `mentora/retrieval/__init__.py` | 新 Django app |
| `mentora/retrieval/apps.py` | AppConfig |
| `mentora/retrieval/jieba_dict.txt` | 55 条自定义术语词典 |
| `mentora/retrieval/tokenizer.py` | jieba 精确模式分词、tsquery 构建、trgm 查询构建 |
| `mentora/retrieval/search.py` | 混合检索：FTS（TF 加权，w=0.7）+ Trgm（字符重叠率模拟，w=0.3）→ RRF 融合 |
| `mentora/retrieval/benchmark.py` | 金标集（7 Evidence + 5 查询）+ P@5/P@10/Recall 评估 |

### 检索流程

```
用户查询
  → jieba 分词（自定义词典确保术语不被切分：「直接映射」→ 整词）
  → FTS 路：词条在 Evidence.content 中的 TF 加权匹配（w=0.7）
  → Trgm 路：字符集重叠率模拟 pg_trgm similarity（w=0.3，阈值 0.15）
  → RRF（k=60）融合 → 按分数降序 top_k
```

## M5-A 追加：Evidence ORM 持久化 + pgvector 配置（2026-06-13）

完整实现检索模块的 Django ORM 模型和 pgvector 基础设施。

### 四个 ORM 模型

按架构文档「单一证据事实 + 多种派生投影」设计：

| 模型 | 表名 | 用途 | 向量 |
|---|---|---|---|
| `EvidenceUnit` | `retrieval_evidence_unit` | 检索事实源，稳定内容/页码/坐标/来源版本 | 无 |
| `ChunkProjection` | `retrieval_chunk_projection` | 聚合相邻 Evidence 为 RAG 文本块 | 全量生成（1536d） |
| `PageTextProjection` | `retrieval_page_text_projection` | 整页文本定位 | 无（仅 FTS） |
| `SentenceProjection` | `retrieval_sentence_projection` | 精确引用和局部查找 | 按需生成 |

### 索引策略

| 模型 | 索引 | 类型 |
|---|---|---|
| EvidenceUnit | `evidence_srcver_page_idx` | B-tree（作用域 + 页码） |
| EvidenceUnit | `evidence_content_trgm_idx` | GIN + pg_trgm（模糊匹配） |
| ChunkProjection | `chunk_srcver_idx` | B-tree（作用域过滤） |
| ChunkProjection | `chunk_embedding_ivfflat_idx` | IVFFlat + cosine（向量检索） |
| PageTextProjection | `page_text_search_idx` | GIN + tsvector（全文检索） |
| SentenceProjection | `sentence_evidence_idx` | B-tree（证据关联） |
