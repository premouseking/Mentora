## 1. 产品与工程目标

SmartStudy 以“每门课程一个独立工作台”为核心。用户主要看到“继续学习”
这一条主线，后台则维护严格、可审计的资料处理、需求澄清、计划生成、学习、
评测和计划调整流程。

架构遵循以下规则：

1. 不根据课程名称自动绑定官方知识库，由用户主动选择参考资料。
2. 官方资料、用户上传资料和用户采纳的网络资源统一建模为可复用资料版本，课程
   通过知识作用域显式选择。
3. 大模型负责组织、解释和个性化表达，不作为事实来源。
4. 新增资料只分析受影响的知识主题，不做所有长文档之间的两两全文比较。
5. 用户可以随时调整学习目标，旧计划保留版本，当前计划按影响范围局部重算。

## 2. 初步复用与提取策略

### 参考学习系统

适合提取或改造：

- Django、Redis、Celery、PostgreSQL 的部署经验；
- MinerU 调用、轮询、重试和 OCR 决策机制；
- PDF 书签、页面结构和句子坐标提取算法；
- 任务快照、心跳和失活任务恢复机制；
- Markmap、ReactFlow、ELK 的可视化实现经验；
- 网页、视频、论文搜索的提供方适配经验。

不直接复用：

- 过于宽泛的 `Resource`、`PDFDocument` 模型；
- 高耦合的 PDF 页面和知识画布；
- 混合了多种业务副作用的全局 Signals；
- 固定维度且直接截断向量的 Embedding 处理；
- 依赖模型输出字符串标记和复杂正则的引用协议。

### 参考桌面智能体

适合提取小型运行时内核：

- 模型与工具的流式事件思想；
- Tool Registry 和有迭代上限的工具调用循环；
- 取消信号、并行只读工具和等待用户回调；
- 搜索结果 URL 归一化、去重与取消机制。

在仓库所有权和许可证未确认前，不直接复制源文件。

## 3. 初步运行架构

```text
React 课程工作台
  -> Django REST API
  -> SSE：对话流、后台进度和用户通知
  -> 应用服务
     -> 显式学习流程状态机
     -> Celery 后台任务
  -> PostgreSQL + pgvector
  -> Redis
  -> COS / MinIO 对象存储
```

第一阶段采用模块化单体。各领域保持明确边界，但共享一个后端部署和数据库。
只有在负载、故障隔离或团队所有权确实要求时才拆分微服务。

当前不预先确定必须使用 LangGraph。应先明确流程状态、持久化和中断契约，
再比较自研状态机、LangGraph 或其他工作流方案。

第一版不引入 Django Channels。用户命令使用普通 HTTP，服务端事件使用可恢复
SSE。只有实时语音、多人协作或浏览器工具回传成为确定需求时再增加 WebSocket。

## 4. 领域模块

### courses

负责课程工作台、模型提取候选、可编辑课程画像、版本化课程知识作用域以及画像与计划
的最终启动。学习意图可随时修改，例如从“长期学习”改为“考前速通”。具体资料选择
不混入课程画像。

### knowledge

历史名称。现行方案拆为 `sources`、`parsing`、`evidence`、`topics` 和
`retrieval`：资源库负责资料身份与版本，课程作用域负责授权，其他模块分别负责
解析、证据、主题和检索。

资料角色不是固定的全局模式，而是某门课程中的动态属性：

- `primary`：当前范围和重点的主要依据；
- `auxiliary`：补充解释、例子和不同视角；
- `supplement`：可选拓展资料；
- `diagnostic`：用于暴露薄弱点的题目、错题或学习记录。

### learning

负责版本化学习路径、阶段/单元/任务模板、计划校验、学习会话、当前任务、学习资源
版本、知识地图、掌握度和复习调度。

建课采用双阶段确认：

```text
结构化澄清
  -> AI 填充 CourseProfileRevision 草稿
  -> 用户修改并点击“生成学习方案”
  -> 冻结 confirmed 画像
  -> 生成可编辑 LearningPlanRevision 草稿
  -> 用户调整路径卡片并通过校验
  -> 点击“开始学习”
  -> 原子激活画像与计划
```

模型的 `extraction_confidence` 只保存在 `CourseProfileCandidate`，用于候选排序和提取
评测。用户确认后的 `CourseProfileField` 保存权威来源，不再保存模型置信度。

### assessment

负责题目、作答、评分规则、错误类型和评测证据。评测不仅包括章节自测，还
包括提问行为、解释能力、间隔回忆、实践任务和知识迁移。

题目采用统一版本化资产模型。平台官方题、用户上传资料提取题、用户手工题和 AI
生成题共享 `AssessmentItemRevision`、审校、组卷和评分协议。日常个性化练习以
AI 动态出题为主：

```text
学生状态和目标误区
  -> 固定课程作用域与证据快照
  -> 先生成 AnswerPlan
  -> 生成题干、干扰项、解析和引用
  -> Schema 与确定性评分检查
  -> 独立盲解、歧义攻击和题型硬验证
  -> 通过后进入当前练习
  -> 根据作答、投诉和统计隔离或复用
```

普通动态题不等待人工审核；验证失败时重生成、回退可靠题或减少题量。人工用于公共
题库、认证等明确高风险用途、异常处理和自动门禁抽样评估。章节练习允许通过完整自动
门禁的题以受限权重使用；模拟考试采用更严格的自动门禁并在开考前完成预组卷，优先混入
人工验证或已校准锚定题，但不要求每道题人工审批。完整设计见
`docs/architecture/assessment-item-bank-design.md` 和
`docs/architecture/adaptive-ai-question-generation-design.md`。

### workflow_runtime

负责流程状态、澄清中断、模型和工具调用、取消、恢复、输出校验及流式事件。

### task_control

负责后台任务租约、心跳、进度、重试、取消、超时和失活任务恢复。

### model_gateway

负责模型注册、硬能力过滤、任务路由、Fallback、Token 预算、调用审计和成本
账本。业务模块只声明任务需要的能力和预算，不直接拼接具体提供方逻辑。

## 5. 文档处理

```text
上传
  -> 文件指纹与重复检测
  -> 创建 SourceVersion
  -> 保存原始文件
  -> 选择解析器
  -> 结构化元素与坐标提取
  -> OCR / 解析质量评估
  -> 语义单元构建
  -> 主题和知识信号提取
  -> 主题对齐
  -> 全文与向量索引
  -> 更新主题证据池
  -> 局部冲突与影响分析
```

PDF 首选 MinerU，但必须通过统一解析器接口接入。解析结果至少保留：

- 资料版本和解析器版本；
- 页面、标题层级和阅读顺序；
- 文本、公式、表格、图片等元素类型；
- 原始文本偏移和 PDF 矩形坐标；
- 跨页关系；
- 解析警告和置信度；
- 原始解析产物引用。

OCR 是否开启不能只依赖“是否存在文本层”，还应综合：

- 有效文本页比例；
- 文本能否反向定位到页面；
- 乱码和替换字符比例；
- 语言字符分布；
- 图片页、扫描页和混合文档比例。

## 6. 检索与证据

初步采用 PostgreSQL 全文检索和 pgvector 的混合检索，并在候选集合上进行
轻量重排。

排序因素包括：

- 语义与关键词相关度；
- 资料在当前课程中的角色；
- 用户明确设置的适用范围和有效时间；
- 解析质量；
- 当前学习目标；
- 主题对齐置信度；
- 来源质量和用户反馈。

每个主题维护证据池。新增资料时先定位受影响主题，再比较这些主题中的结构化
知识主张，而不是比较整篇文档摘要。

检索数据采用“单一证据事实 + 多种派生投影”：

- `EvidenceUnit` 保存稳定内容、页码、矩形、结构类型和来源版本；
- `ChunkProjection` 聚合证据单元，承担默认 RAG；
- `PageTextProjection` 和 `PageImageProjection` 服务整页定位与视觉检索；
- `SentenceProjection` 服务精确引用和局部查找。

默认只全量生成 Chunk 文本向量。句子位置全量保存，但句子向量按需求生成；
页面图像向量只对扫描件、图表密集资料或开启视觉检索的课程生成。展示用引用
标记不生成第二份向量。

资料范围分为三层：

- 课程 `CourseKnowledgeScopeRevision` 中长期启用的资料版本；
- 本次学习会话 `SessionScopeOverlayRevision` 临时启用或排除的资料版本；
- 单次模型调用实际使用的不可变证据快照。

历史回答始终保留原作用域修订、资料版本和证据快照，新资料只影响后续调用。作用域
采用完整 Binding 快照运行，差异记录只用于影响分析和审计；恢复历史配置时创建新的
修订，不直接重新激活旧版本。

冲突等级：

- `P0`：改变考试范围、关键结论或学习计划，必须澄清；
- `P1`：改变重点或优先级，展示推荐处理方式；
- `P2`：记录差异，不中断当前学习；
- `P3`：低置信噪声，暂不处理。

## 7. 学习流程

```text
解析初始诉求
  -> 判断画像字段是否足够
  -> 必要时提出少量高价值问题
  -> AI 填充课程配置面板
  -> 用户修改并确认画像
  -> 固定计划输入版本
  -> 创建可编辑路径草稿
  -> 校验前置、预算、覆盖和资料可用性
  -> 用户最终确认并启动
  -> 生成当前学习包
  -> 学习 / 提问 / 练习
  -> 收集掌握度证据
  -> 更新掌握度与置信度
  -> 选择下一项任务
```

流程状态只保存标识符、版本、决策和结果引用，不把长文档正文塞入状态。

建课状态为：

```text
collecting_requirements
  -> reviewing_profile
  -> generating_plan
  -> reviewing_plan
  -> active
```

计划卡片允许修改独立单元顺序、任务类型和预算内时间分配。修改截止日期、总时间预算、
范围或目标深度时必须回到课程画像，旧计划草稿进入 `needs_replan`。最终启动事务必须
同时切换 `active_profile_revision_id` 和 `active_learning_plan_revision_id`。

模型上下文由服务端统一预算和组装。前端只表达用户选择的资料、顺序和操作
意图，不直接负责拼接最终 Prompt。

## 8. 上下文账本与模型视图

服务端同时维护两套不同语义的数据：

- `ContextLedger`：目标变更、资料选择、学习事件、澄清结论和工作流决策的
  事实账本；
- `ContextView`：针对某次任务和模型，在 Token 预算内生成的临时上下文。

每次目标、资料范围或学习策略改变时创建新的 `ContextRevision`。首次模型调用
注入完整基线，后续调用优先追加版本差异，避免重复发送稳定信息。

当历史达到模型预算阈值或学习状态发生重要变化时，后台生成结构化
`LearningSnapshot`。产品层可以称其为学习摘要，但领域层不再维护单一自由文本
`LearningSummary`。快照至少保留：

- 当前目标和计划版本；
- 课程知识作用域版本；
- 已完成主题和掌握度变化；
- 已识别误区；
- 未解决问题；
- 用户偏好；
- 证据和近期交互引用。

摘要校验成功后才切换活动模型视图，原始学习事件和证据不删除。模型上下文由
“结构化摘要、最近原始交互、当前证据快照”组成。

学习记忆进一步拆为四类对象，避免把不同语义混入一段文本：

- `LearningEvent`：不可变学习事件；
- `LearnerFact`：带来源、置信度和有效期的用户/课程事实；
- `MasteryEvidence`：知识点掌握证据；
- `LearningSnapshot`：带事件水位和生成版本的派生快照。

课程快照在会话结束、有效事件达到阈值、目标变化、资料影响当前主题或上下文
接近预算时增量生成。生成失败时不推进事件水位，旧快照保留用于回放和质量
对比。掌握度只能由 `MasteryEvidence` 更新，不能从聊天摘要或用户画像直接推断。

资料采用渐进式加载：先向模型提供资料目录元数据，再按当前任务检索证据正文。
解析、搜索和测试的完整长输出进入对象存储或结果表，模型只看到统计信息、首尾
片段和可继续读取的 Artifact ID。

## 9. 运行时事件与前端同步

聊天 Token、工作流阶段、解析进度和任务通知统一为 `RuntimeEvent`。每个事件
具有稳定 `event_id`、流 ID 和单调递增序号。

前端通过 SSE 订阅，断线后从最后事件序号恢复；回放窗口失效时重新读取 REST
任务快照。事件推送不是事实源，领域状态和最终产物必须先持久化。

用户执行澄清回答、取消、调整目标和提交测试时使用 HTTP 命令接口。SSE 断开
不会取消后台任务，取消必须由显式命令驱动状态机。

## 10. 有依据的内容生成

内容按当前主题、当前任务按需生成，不为整门课程提前生成所有个性化内容。

一个学习包可以包含：

- 简明讲解；
- 教材或资料中的依据；
- 例题和推导；
- 针对性练习；
- 易错点；
- 可选网络资源。

生成产物必须记录：

- 资料版本；
- 证据单元 ID；
- 生成模型和 Prompt 版本；
- 个性化参数；
- 自动质量检查结果；
- Token 和费用；
- 失效原因与影响范围。

引用应作为结构化字段返回，不能主要依赖大模型在正文中拼接特殊字符串。

## 11. 学习进度与掌握度

记录查看、提问、作答、解释、完成、复习和实践等学习事件。

每个主题维护：

- 掌握度估计；
- 估计置信度；
- 正反证据数量；
- 最近一次有效评测；
- 遗忘风险；
- 下次复习时间。

不能把“看完页面”直接等同于掌握。进度由多种证据共同决定：

- 无提示回忆；
- 不同难度题目的表现；
- 错误类型是否重复；
- 能否解释原因；
- 是否能迁移到新问题；
- 隔一段时间后是否仍然会做；
- 提问是否从基础事实转向边界和应用。

## 12. 用户体验

课程工作台的主操作是“继续学习”。次级入口包括上传资料、提问、测试、调整
目标和查看知识地图。

知识地图必须是可操作的主题图，而不是装饰性图谱。节点至少展示掌握度、
置信度、重要性、资料覆盖、前置关系和下一步操作。

用户可以随时改变学习目标。系统应展示变化影响，例如：

- 哪些任务被提前或延后；
- 哪些已有资源仍然有效；
- 哪些内容需要重新生成；
- 是否出现新的资料冲突；
- 预计时间如何变化。

## 13. 安全、质量与成本

- 每个课程和资料查询都校验用户所有权；
- 文件扩展名和真实类型分别校验，并进行恶意文件扫描；
- 对象存储使用签名 URL；
- 未抓取、未解析的网络推荐不能直接作为事实依据；
- 模型日志中移除不必要的个人数据；
- 追踪请求、流程、任务、模型调用和检索链；
- 评估召回率、引用覆盖、依据一致性、事实正确性、题目质量、延迟和成本；
- 为课程、用户、流程和模型设置费用预算与降级策略。
- 将模型请求流量与实际 Token 用量分开记录，失败请求也可追踪；
- 记录请求模型和实际模型，避免 Fallback 后无法解释质量与成本变化。
- 监控上下文压缩前后 Token、关键信息保持率、证据引用保持率和压缩失败率；
- 长输出裁剪后仍保留完整 Artifact，避免无法审计模型遗漏了哪些内容。
- 评估学习快照的目标、未决问题、错误概念和证据引用保持率；
- 驱动计划或掌握度变化的摘要事实必须带原始事件或证据引用。

模型调用账本拆分为：

- `ModelRequest`：一次逻辑任务及其路由要求；
- `ModelAttempt`：每一次实际网络调用，包括重试和 Fallback；
- `UsageSettlement`：内部成本、用户费用、退款和对账结果。

失败 Attempt 不删除。已知上游消耗计入内部成本，未知用量进入待对账状态；
用户只按产品结算规则扣费。流式输出一旦对用户可见，不允许静默从头重试。

## 14. 交付阶段

1. 资料闭环：课程、资料版本、解析、检索、有引用的问答。
2. 学习闭环：意图澄清、计划、学习会话、评测和掌握度。
3. 资料演进：新增资料影响分析、冲突澄清和局部计划调整。
4. 丰富资源：知识地图、思维导图、外部推荐和练习资源。
5. 工程强化：离线评测、可观测性、配额、成本控制和负载测试。

## 15. 关键技术选型结论

### 15.1 工作流运行时

首版使用 Django + Celery + PostgreSQL 的持久状态机，不直接引入 Temporal 或
LangGraph 作为业务事实源：

- 状态迁移在数据库事务内提交；
- Outbox 负责可靠发布运行时事件；
- 每次外部副作用都有 Attempt 和幂等键；
- 澄清通过持久化 `ClarificationRequest` 暂停并恢复；
- Worker 崩溃后根据租约、心跳和状态重新调度；
- 重试针对活动步骤，不从头静默重放整个用户流程。

Temporal 作为后续升级候选；LangGraph 只允许用于边界清晰的 Agent 子流程。

### 15.2 检索存储

首版优先使用 PostgreSQL + pgvector：

- PostgreSQL 保存资料、版本、权限、证据和索引元数据；
- 关键词与向量检索必须应用同一组结构化过滤；
- 两路候选默认使用 RRF 融合；
- 高风险回答可增加 cross-encoder 或 LLM reranker；
- 中文关键词实现保留适配器，通过真实资料基准后再确定 `pg_trgm`、N-gram、
  外部分词 `tsvector` 或稀疏向量方案。

只有当数据量、吞吐量或检索能力证明单库无法满足要求时，才引入独立 OpenSearch
或向量数据库，并同时设计变更日志、重建和一致性对账机制。
