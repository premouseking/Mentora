# 四人团队工程章程

## 1. 交付目标

第一个产品里程碑是跑通一条完整、基于资料证据的学习闭环：

```text
桌面端登录
  -> 上传文本 PDF
  -> 将资料加入课程
  -> 解析并检索证据
  -> 生成带页码引用的回答
  -> 确认课程画像
  -> 生成并确认学习计划
  -> 完成一次学习任务
  -> 完成一次基础测验
```

在这条链路稳定之前，团队不优先扩展更多资料格式、高级测评、离线能力或微服务。

## 2. 职责模式

每个模块只有一位直接责任人（DRI）。其他成员可以参与开发，但 DRI 必须对
接口清晰度、测试、文档、异常处理和最终可运行结果负责。

### LBZ：产品与前端

负责：

- `apps/web/`；
- 课程列表、建课、画像确认、计划确认、课程工作台、学习、Tutor、测评、
  原文查看和工作流进度界面；
- React Query 服务端状态管理规范；
- 无障碍、响应式布局和用户可理解的错误恢复；
- Renderer 组件测试和端到端用户流程。

禁止：

- 在前端拼装最终模型 Prompt；
- 在 Renderer 中保存权威课程状态或掌握度；
- 访问 Node.js、任意本地文件、Token 或外部模型服务；
- 使用长期 Mock 数据伪装已经完成后端集成。

每个功能必须交付：

- 路由和完整界面状态；
- 类型化请求和响应；
- 加载、空数据、失败、重试和成功状态；
- 组件测试或流程测试；
- 面向用户的验收演示。

### WH：组长、后端与平台

负责：

- 项目总体架构、阶段范围、任务协调和最终集成验收；
- `apps/api/config/`；
- 课程、画像、计划、工作流、运行时事件、认证和授权服务；
- PostgreSQL 迁移、Redis、Celery、MinIO 和本地开发基础设施；
- 后续 `apps/desktop/` 的 Main 和 Preload；
- 事务、幂等、并发、审计、API 和 SSE 规范。

必须保证：

- PostgreSQL 是业务事实源；
- Celery 只执行任务，不拥有业务事实；
- 画像和计划的启用具有原子性；
- 重试不会生成重复版本、任务或事件；
- SSE 可以断线续传，REST 可以恢复最终状态；
- 跨领域写操作通过应用服务完成。

每个功能必须交付：

- 数据迁移和领域服务；
- API 契约和统一错误格式；
- 状态迁移与权限测试；
- 幂等和并发行为；
- 日志及故障恢复方式。

### LH：资料与检索

负责：

- 与 B 共同定义资料身份和不可变资料版本；
- 解析、`ParsedBundle`、`EvidenceUnit`、检索投影、Embedding 和召回；
- 对象存储产物和处理任务；
- 课程作用域过滤和原文引用定位；
- 解析与检索基准数据集。

必须保证：

- 上传资料不等于授权课程使用；
- 服务端只在课程当前作用域内检索；
- 历史回答保留资料、证据和作用域快照；
- 调整切块或 Embedding 策略不会改变原始证据；
- 重复处理使用稳定的幂等键；
- 先完成文本 PDF，再扩展 OCR、Office、音频和视频。

每个功能必须交付：

- 带版本的 Schema 和解析输出；
- 基准测试数据和质量指标；
- 明确的失败阶段和重试语义；
- 资料页码及坐标定位；
- 延迟和检索质量报告。

### LWJ：AI 学习与评测

负责：

- 模型网关、路由、超时、Fallback、用量账本和 Prompt 版本；
- 课程画像候选提取和需求澄清；
- 学习计划和 Tutor 回答生成；
- 阶段总结和计划调整建议；
- 测评题生成、确定性检查、独立盲解、歧义检查和离线评测。

必须保证：

- 模型只产生候选，不直接产生权威业务事实；
- 所有结构化输出经过 Pydantic 或 JSON Schema 校验；
- 模型只能接收服务端批准的证据快照；
- 出题模型和独立验证模型不能共享答案；
- 质量门失败时拒绝或降级，不能把不确定题目展示给用户；
- 模型、Prompt、证据、成本和延迟均可追踪。

每个功能必须交付：

- 输入输出 Schema；
- Prompt 和策略版本；
- 失败及降级行为；
- 金标集测试和评测报告；
- 模型调用审计字段；
- 成本和延迟预算。

## 4. 接口所有权

依赖开发开始前，以下契约必须在仓库中完成版本化：

| 契约 | 负责人 | 必须参与评审 |
| --- | --- | --- |
| Desktop API 和 IPC Schema | WH | LBZ |
| REST 错误和 Runtime Event | WH | LBZ、LH、LWJ |
| Source、ParsedBundle、EvidenceUnit | LH | WH |
| RetrievalRequest 和 EvidenceSnapshot | LH | LWJ、WH |
| 画像和学习计划 DTO | WH | LBZ、LWJ |
| 模型任务输入输出 Schema | LWJ | WH、LH |
| 测评题目和 Attempt DTO | WH | LBZ、LWJ |

破坏兼容性的契约修改必须同时包含：

1. 决策记录；
2. 所有消费者评审；
3. 数据迁移或兼容性说明；
4. 同一个变更中的契约测试。

## 5. 任务状态

```text
待分析 -> 可开始 -> 开发中 -> 评审中 -> 已集成 -> 已验收
```

- `待分析`：有价值，但信息不足，不能开始。
- `可开始`：满足 Definition of Ready。
- `开发中`：一位 DRI 正在处理。
- `评审中`：代码和验收证据已经提交。
- `已集成`：已经与上下游连接。
- `已验收`：按照验收标准完成演示。

每个人同时最多有一个主要任务处于“开发中”。

## 6. Definition of Ready

任务开始前必须具备：

- 一个用户可见或系统可观察的结果；
- 一位 DRI 和明确的评审人；
- 清楚的范围和不做事项；
- 输入和输出契约；
- 依赖与阻塞项；
- 可观察的验收标准；
- 明确的测试方式；
- 预计修改的文件或模块；
- 不存在尚未解决的架构冲突。

创建任务时使用[任务模板](templates/task-template.md)。

## 7. Definition of Done

任务只有满足以下条件才算完成：

- 实现已经合并或达到可合并状态；
- 相关单元、契约、集成或端到端测试通过；
- 需要时已覆盖失败、重试、取消和权限行为；
- 正式路径中不存在永久 Mock；
- Schema、文档与实现一致；
- 日志不包含 Token、资料正文、隐藏答案或预签名 URL；
- DRI 已按照验收标准演示；
- 下游负责人确认接口可以正常消费。

异步任务还必须具备幂等、超时、有限重试和失活任务恢复。

AI 任务还必须具备 Schema 校验、金标集评测、降级行为、Prompt 版本以及
成本和延迟报告。

## 8. Pull Request 规则

- 每个 PR 只处理一个行为或一项契约变更。
- 修改共享契约时，所有契约评审人必须参与。
- 数据库迁移必须由 B 评审。
- 检索和证据变更必须由 C 评审。
- Prompt、模型策略或 AI Schema 变更必须由 D 评审。
- 用户流程和 Renderer 行为必须由 A 评审。
- 作者不能是唯一执行用户流程验收的人。
- 检查失败或破坏性变更未记录时禁止合并。

## 9. 团队节奏

### 阶段启动会

- 确定当前阶段唯一的集成目标；
- 只把满足 Ready 的任务放入当前阶段；
- 找出关键依赖链；
- 确认每个人的主要任务。

### 每日同步，15 分钟

每个人只说明：

1. 已完成的可验证结果；
2. 今天要交付的具体产物；
3. 当前阻塞项及负责人；
4. 是否存在接口变更风险。

### 阶段中集成检查

- 运行当前端到端链路；
- 解决接口不一致；
- 删除或延期威胁阶段目标的非必要范围；
- 记录技术决策。

### 阶段验收与复盘

- 从干净环境演示用户流程；
- 检查未通过的验收项；
- 记录一项流程改进；
- WH 根据退出标准决定是否进入下一阶段；
- 更新路线图风险和下一阶段任务清单。

## 10. 升级规则

- 阻塞超过一个工作日，必须在团队渠道中明确负责人和决策期限。
- 接口争议由接口负责人在咨询所有消费者后决策，并记录结果。
- 安全或数据一致性风险出现时，立即停止发布。
- AI 输出质量问题不能通过无评测依据地降低硬门槛解决。
- 阶段目标有风险时，优先删除可选范围，不能把端到端链路拆成几套无法集成的半成品。
